New Project Kickoff

Foundation only — local folders, repo, template clone, lead setup. Stops before presearch.

Template version: v0.2.2  ·  For: Week 4 hiring-partner project  ·  Time budget: ~20 minutes

"Partner project" = work for a real hiring partner. The deliverable goes to an external stakeholder who may have IP, license, NDA, security, or repo-hosting requirements that override the defaults below. Confirm Step 0 before you create anything.

2Create GitLab repo (submission target) + clone template

GitLab is the primary repo for Gauntlet submission and CI. GitHub's "Use this template" button would create the project under GitHub's namespace — wrong primary. Instead: clone the template's working tree, re-init pointing at GitLab, push.

2a — Create the empty GitLab repo

glab repo create coryvandenberg/<new-project-name> \
  --private \
  --description "<one-line description>"

Or via web UI: labs.gauntletai.com/projects/new  ·  namespace coryvandenberg, private, no README, no .gitignore, no license (the template provides those).

2b — Pull the template's working tree (no template git history)

cd C:/Dev/GauntletAI/
git clone --depth=1 --branch v0.2.2 \
  https://github.com/TradeUpCards/claude-code-project-template.git \
  <new-project-name>
cd <new-project-name>
rm -rf .git
git init -b main
Why --depth=1 --branch v0.2.2 + delete .git. Pins to the v0.2.2 tag so template churn on main won't surprise your build. Deleting .git drops the template's commit history so your new project starts with one clean commit, not the template's commits.

3Wire dual-push: GitLab primary + GitHub backup conditional

Skip this step if the hiring partner hasn't explicitly OK'd a personal-account backup mirror. For hiring-partner work, mirroring confidential code to a personal GitHub — even private — can violate NDA or IP terms. Default for partner work is GitLab single-remote. Just run: git remote add origin https://labs.gauntletai.com/coryvandenberg/<name>.git and skip to Step 4.

If the partner OK'd the backup mirror (or there's no NDA): this is the pattern from AgentForge-tate — one origin remote that pushes to both. Fetch from GitLab only; push updates both.

3a — Optional but recommended: create the GitHub backup repo

gh repo create TradeUpCards/<new-project-name> \
  --private \
  --description "Backup mirror of GitLab <new-project-name>"

3b — Configure dual-push origin

# Add GitLab as fetch + first push target
git remote add origin https://labs.gauntletai.com/coryvandenberg/<new-project-name>.git

# Add GitHub as a SECOND push URL on the same origin remote
git remote set-url --add --push origin https://labs.gauntletai.com/coryvandenberg/<new-project-name>.git
git remote set-url --add --push origin https://github.com/TradeUpCards/<new-project-name>.git

# Verify — should show 1 fetch, 2 push URLs
git remote -v

Expected output:

origin  https://labs.gauntletai.com/coryvandenberg/<name>.git  (fetch)
origin  https://labs.gauntletai.com/coryvandenberg/<name>.git  (push)
origin  https://github.com/TradeUpCards/<name>.git              (push)
Workflow rule (from AgentForge-tate memory). Merge MRs on GitLab only, then run git push origin locally to update both. Never accept a PR on GitHub — that creates permanent SHA divergence between the two repos. If you forget and the GitHub mirror drifts, recover with: git fetch origin master && git push origin refs/remotes/origin/master:refs/heads/master

Hiring-partner project specific. The partner is the customer, not a teammate — they don't have commit rights or sit in Cursor with you. The decisions that shape in-flight.md are: which workstreams map to which lead, and which sessions can run in parallel without same-file collisions. Use the partner's PRD / spec to anchor lead workstreams; if the partner hasn't issued one, /presearch-interview + /weekly-prd in Step 8 fills that gap.
Where .project/ lives (it's not in the repo). The .project/ directory — kickoff prompts, lead handoffs, session summaries, story candidates, coordination threads — is gitignored by default in this template (verified in all three gitignore options). It's local-only, mirrored to OneDrive as the survivor copy. It never ships to GitLab, never ships to a partner submission, never appears in commit history. So partner-confidentiality concerns about commit history don't apply to .project/ — the only question is whether your OneDrive personal cloud is acceptable backup storage for partner work artifacts. If unclear, point the OneDrive mirror at a partner-approved location instead (or skip mirroring entirely and accept the survivor-copy gap).
