# Story candidates ‚Äî backlog of moments to capture

> Working file. Add a row when a story-worthy moment happens; promote to a
> full story via `/story`. When you write the full version, delete the row
> here (or mark `[WRITTEN]`).
>
> Don't over-curate. The 6th candidate is the one you'll forget without
> writing down. The 14th captured story is what makes a portfolio look real.

---

## How to add a candidate

One row per moment. Six columns:

| Column | What goes here |
|---|---|
| **Date** | YYYY-MM-DD when the moment happened |
| **Category** | `architecture` / `security` / `scope` / `incident` / `meta` ‚Äî see taxonomy below |
| **Shape** | `design-defense` / `STAR` / `war-story` / `scope-decision` (story-template type, distinct from category) |
| **Hook seed** | One-line headline you'd rewrite later |
| **Why it matters** | Audience: grader / CTO / employer / clinician ‚Äî and what it demonstrates |
| **Min facts to recover** | The 2‚Äì3 specifics you'd forget in a week (file:line, commit, number, decision moment) |

### Category taxonomy

| Category | What it covers | Filter use |
|---|---|---|
| **architecture** | Technical design choices defended, type-system uses, structural decisions | Engineer interview, system design Q |
| **security** | Sensitive-data handling, threat-model, audit findings, defense-in-depth | CTO / compliance / security reviewer Q |
| **scope** | Sizing decisions, deferrals, pivots, estimate revisions, what was cut | PM / project-management Q |
| **incident** | Bug or failure caught and fixed, with diagnostic chain | "Tell me about a time you debugged" Q |
| **meta** | Tooling, workflow, process, AI-collaboration patterns | "How do you work with AI tools" Q |

A story can fit two categories ‚Äî pick the dominant one (the audience-fit you most want this story to surface for). The shape column is independent: a `meta` story can be a `design-defense` shape if it defends a process choice, or a `STAR` shape if it walks through what happened.
